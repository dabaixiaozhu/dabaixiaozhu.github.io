# <a name="2">[客户端与服务器端](#begin)</a>

## <a name="2-1">[客户端](#begin)</a>

- 生产者：Producer。向主题发布新消息的应用程序
- 消费者：Consumer。从主题订阅新消息的应用程序
  - 消费者位移：Consumer Offset。表征消费者消费进度，每个消费者都有自己的消费者位移
  - 消费者组：Consumer Group
    - 多个消费者实例共同组成的一个组，同时消费多个分区以实现高吞吐
    - 这组主题中的每个分区都只会被组内的一个消费者实例消费，其他消费者实 例不能消费它
  - 重平衡：Rebalance
    - 消费者组内某个消费者实例挂掉后，其他消费者实例自动重新分配订阅主题分区的过程
    - 是消费者端实现高可用的重要手段

## <a name="2-2">[服务器端](#begin)</a>

- kafka cluster
  - 一 个 Kafka 集群由多个 Broker 组成，Broker 负责接收和处理客户端发送过来的请求，以及对消息进行持久化		

  - 将不同的 Broker 分散运行在不同的机器上，这样如果集群中某一台机器宕机，即使在它上面 运行的所有 Broker 进程都挂掉了，其他机器上的 Broker 也依然能够对外提供服务

- 每个Broke包括的内容
  - 消息：Record。指 Kafka 处理的主要对象
  - 主题：Topic。是承载消息的逻辑容器，在实际使用中多用来区分具体的业务
  - 分区：Partition
    - 特点
      - 是一个有序不变的消息序列，每个主题下可以有多个分区
      - 生产者生产的每条消息只会被发送到一个分区中
    - 消息位移：Offset
      - 生产者向分区写入消息，分区中每条消息的位置信息
      - 是一个单调递增且不变的值，总是从 0 开始
    - 副本：Replica
      - 在分区层级下的，即每个分区可配置多个副本实现高可用
      - 区分
        - 领导者副本（Leader Replica）：者对外提供服务，即与客户端程序进行交互
        - 追随者副本（Follower Replica）：是被动地追随领导者副本而已，不能与外界进行交互
      - 工作机制
        - 生产者总是向领导者副本写消息；消费者总是从领导者副本读 消息。
        - 追随者副本向领导者副本发送请求，请求领导者把最新生产的消息发给它，这样它能保持与领导者的同步

# <a name="3">[Broker三层消息架构](#begin)</a>

- 1主题层
  - 每个主题可以配置 M 个分区
  - 每个分区又可以配置 N 个副本

- 2分区层
  - 每个分区的 N 个副本中只能有一个充当领导者角色，对外提供服务；
  - 其他 N-1 个副本是追随者副本，只是提供数据冗余之用 

- 3消息层
  - 分区中包含若干条消息，每条消息的位移从 0 开始，依次递增
  - 客户端程序只能与分区的领导者副本进行交互

# <a name="4">[Broker持久化数据](#begin)</a>

Kafka 使用 消息日志（Log）来保存数据，因为只能追加写入，故避免了缓慢的随机 I/O 操作，改为性能较好的顺序 I/O 写操作

定期删除消息以回收磁盘

- 通过日志段（Log Segment）机制：一 个日志又近一步细分成多个日志段，消息被追加写到当前最新的日志段中，当写满了一个日 志段后，Kafka 会自动切分出一个新的日志段，并将老的日志段封存起来
- Kafka 在后台还 有定时任务会定期地检查老的日志段是否能够被删除，从而实现回收磁盘空间的目的
